## What Shlink Actually Is

[Shlink](https://shlink.io/) is a self-hosted, open-source URL shortener. You point it at your own domain, and it gives you short codes for long URLs, click analytics (referrers, devices, rough geolocation), custom slugs, and multi-domain support — the stuff Bitly charges for, running on hardware you already own.

Normally Shlink expects you to live in two places: the CLI (`bin/cli`) for admin tasks like generating API keys, and the REST API for everything else. There's also an official **web client** that talks to the API for you, which is the piece that makes a no-code setup possible.

> 💡 If you're already running Traefik, Portainer, and a handful of other containers like I am, Shlink slots into the stack the same way everything else does — one more compose file, one more subdomain.

## The No-Code Catch (and How to Dodge It)

Here's the part every Shlink guide glosses over: before the web client can do anything, it needs an **API key**. The "normal" way to get one is a CLI command — `shlink api-key:generate` — run inside the container.

That's a dealbreaker if you want zero terminal time. The fix is an environment variable Shlink has supported since version 3.3.0: **`INITIAL_API_KEY`**. Set it to any string you like when you deploy the stack, and Shlink creates that exact key as a valid API key on first boot — no CLI step required. You just typed the key into a form field in Portainer instead of a terminal, which is exactly the kind of "no code" we're going for here.

## Deploying the Stack in Portainer

In Portainer, go to **Stacks → Add stack**, give it a name like `shlink`, and paste this into the web editor:

```yaml
version: "3.8"
services:
  shlink:
    image: shlinkio/shlink:stable
    container_name: shlink
    restart: unless-stopped
    environment:
      - DEFAULT_DOMAIN=links.networkeclipse.com
      - IS_HTTPS_ENABLED=true
      - INITIAL_API_KEY=replace-this-with-your-own-long-random-string
      - DB_DRIVER=sqlite
    volumes:
      - shlink_data:/etc/shlink/data
    ports:
      - "8090:8080"

  shlink-web-client:
    image: shlinkio/shlink-web-client:stable
    container_name: shlink-web-client
    restart: unless-stopped
    environment:
      - SHLINK_SERVER_URL=https://links.networkeclipse.com
      - SHLINK_SERVER_API_KEY=replace-this-with-your-own-long-random-string
      - SHLINK_SERVER_NAME=Homelab
    ports:
      - "8091:8080"

volumes:
  shlink_data:
```

A few notes on what's happening there:

- **`DEFAULT_DOMAIN`** — the public-facing domain your short links will use. Swap in whatever you've pointed at this box.
- **`INITIAL_API_KEY`** — must match exactly between the `shlink` service and the `SHLINK_SERVER_API_KEY` in `shlink-web-client`. Pick a long, random string; this is effectively your admin password.
- **`DB_DRIVER=sqlite`** — keeps this simple for a homelab. Shlink also supports MySQL/MariaDB/Postgres if you'd rather point it at an existing database container.
- Pre-loading `SHLINK_SERVER_API_KEY` into the web client means you skip the "add a server" form entirely — it shows up already connected.

Hit **Deploy the stack** and let Portainer pull the images.

> ⚠️ The web client has no login screen of its own — anyone who can reach it can manage your links. Keep it on your LAN, behind your VPN, or behind a reverse proxy that handles auth (Authelia, etc.). Don't expose port 8091 straight to the internet.

## Connecting the Web Client

Because you pre-loaded the API key via environment variables, there's nothing left to configure. Open `http://your-server-ip:8091` and you should land directly on a dashboard already pointed at your `Homelab` server — no setup screen, no copy-pasting keys around.

If you ever do need to add a second Shlink instance (say, a separate one for a different domain), the **Manage servers** screen in the web client lets you add it with a name, URL, and API key, all through form fields — still zero code.

## Creating Your First Short Link

From the dashboard:

1. Click **Create short URL**.
2. Paste the long URL into **Long URL**.
3. Optionally set a custom slug under **Custom slug** (e.g. `jellyfin` instead of a random code).
4. Optionally set tags, a title, or a "valid until" expiration date.
5. Click **Create**.

Your new link is live immediately at `https://links.networkeclipse.com/your-slug`. Click into any link from the list to see its visit stats — referrers, browsers, rough location — all rendered as charts in the same UI.

## Putting It Behind Traefik

If you're running Traefik v3 the way [I set it up for wildcard SSL across my domains](#), add labels instead of exposing the raw port. Drop this into the `shlink` service (and skip the `ports:` mapping for it):

```yaml
labels:
  - "traefik.enable=true"
  - "traefik.http.routers.shlink.rule=Host(`links.networkeclipse.com`)"
  - "traefik.http.routers.shlink.entrypoints=websecure"
  - "traefik.http.routers.shlink.tls.certresolver=cloudflare"
  - "traefik.http.services.shlink.loadbalancer.server.port=8080"
```

Do the same for `shlink-web-client` on whatever internal subdomain you want it to live on, then lock that one down with split-horizon DNS so it never resolves outside your LAN.

## Wrapping Up

That's the whole stack: one compose file, one environment variable doing the job a CLI command usually does, and a web client that's already connected the moment it starts. From here it's all clicking — create links, check stats, manage servers — no shell required.

If you self-host more than a couple of internal services, having your own short link domain ends up paying for itself fast. It's also just nice to hand someone `links.networkeclipse.com/jellyfin` instead of a 90-character URL with a port number in it.
